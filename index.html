As we can see from the codes above, we can give the machine — as an object — one of certain type: whether the integer number 5 or the string Hello, for example.

Indeed, there are more basic types of objects (than just integer numbers or strings) that we might give to the machine. If wanted, you yourself might do your own searching over this topic on the Internet. Our advise is: Don't do it so far.

We here will only take a look at the 2 basic types of objects:
· Int — which might be represented by the integer number 5, for example
· String — which might be represented by the string Hello, for example.



Of course, the Int here stands for the integer number.

Absolutely, juggling just these 2 basic types of objects — Int and String — are enough to get an idea of the entire basic type concept in programming.

On the Earth, there are lots of stupid programming languages in which programmers always must explicitly write (report, state) the basic type of the object they give to the machine — whether it is Int or String, for example.

Fortunately, in Kotlin and Swift this reporting is only optional.

It is possible since with Kotlin and Swift the machine itself can understand which type of data is it — whether it is Int or String, for example. When the machine do so it is called type inference.

Type inference is done automatically under the hood — in other words, without our participation and only by the machine itself.

Anyways, type inference is one of reasons why people around the world love Kotlin and Swift so much. Type inference makes the coder's life a lot easier. Simultaneously, it makes code more clear and concise, meaning awesome.

Glory to the Type Inference! We mean, glory to the teams behind Kotlin and Swift!

Your Own Slave Commanders
 You already know a lot of awesome things about Commanders:


Each Commander has its name and makes the machine do some specific work.

There are various sorts of Commanders.

You already know a lot about Pocket Commanders, Arithmetic Commanders, the Commander print and others.

Anyways, reality, again, is even more stunning!

You may create your own Slave Commanders:


Each your Slave Commander will be a slave to you but the commander to the machine.

When you create your Slave Commander, you can establish (describe) the specific work that this Slave Commander will make the machine do.

So your Slave Commander can make the machine do precisely the specific work that you previously established (described).

By the way, since all Commanders are our helpers, they all in a way are our slaves. But the thing of Slave Commanders is that precisely you establish (describe) the specific work that your Slave Commander makes the machine do.

Slave Commanders are called Functions.

Creating a Function happens with the word fun in Kotlin and func in Swift.

Let's create a Function named addTwoTo( ) { } that would add the number 2 to any number that we give to the Function.

fun addTwoTo(anyNumber: Int) {

Very Interesting Commander
 We already know that in programming the number 5 or the string Hello, for example, is an object. And we can control these objects — in other words, we can do some action on these objects.

We also know that Commanders — for example, Pocket Commanders or Array Commanders — may be objects as well; and we can also do some action on these objects.

But additionally, we can also create much more interesting objects than all those ones we know for now.

Let's start diving into this unbelievably wonderful world.

Suppose that we want to write a program to record information about pets.


As we continue thinking about them, we realize that all pets are quite similar.

If we take, for example, a dog and a cat; each of them has its name, its age, its breed, its sex, its color, its way they sound, and so on.

In other words, each of them has its object for each its parameter:

species = "dog", name = "Max", age = 3, breed = "Beagle", sex = "male", color = "brown", sound = "woof", class = "pet"

species = "cat", name = "Kitty", age = 2, breed = "Burmilla", sex = "female", color = "white", sound = "meow", class = "pet"

As we can see, they both — the dog Max and the cat Bella — have and will always have one thing in common: each of them is a pet!

So the class pet is what they have in common — in other words, the pet is the object they have and will always have, for their parameter species, in common.

An example of code:
  
class pet(
    var anySpecies: String,
    var anyName: String,
    var anyAge: Int
    ) {
    init {







    println("$anyName is a $anySpecies.")
    println("It is $anyAge.")
    println(" ")       
    }
}
fun main() {
    var d = pet("dog", "Max", 3)
    var c = pet("cat", "Bella", 2)
}
RESULT:
Max is a dog.
It is 3.
 
Bella is a cat.
It is 2.
